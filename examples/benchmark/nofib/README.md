# nofib for PureLang

A port of (part of) the [GHC `nofib` benchmark suite](https://github.com/ghc/nofib)
to **PureLang**, the source language of the PureCake compiler. The original suite
and its "rules of the game" are described in Will Partain's paper, *The `nofib`
Benchmark Suite of Haskell Programs* (`../../../nofib.pdf`).

Each benchmark is a real PureLang program with a recorded workload and expected
output, so the suite doubles as an end-to-end correctness *and* performance test
for the compiler.


## What's here

```
nofib/
  run.py                     # compile + run + check driver
  imaginary/<name>/
    <name>.hs                # the benchmark (one self-contained file)
    sizes.cfg                # fast / norm / slow command-line arguments
    <name>.faststdout        # expected output for `fast`
    <name>.stdout            #   "        "     for `norm`
    <name>.slowstdout        #   "        "     for `slow`
  spectral/<name>/ ...
```

This mirrors `nofib`'s split into **imaginary** (small, self-contained toy
programs) and **spectral** (algorithmic kernels). The much larger **real**
subset is not attempted — those programs rely on a module system, file/stdin I/O
and floating point that PureLang does not (yet) provide.


## How PureLang shapes the port

PureLang is small and the ports are necessarily faithful *in spirit* rather than
line-for-line. The constraints that matter (see `../../syntax.hs`):

- **Only `Integer`.** There is no `Double`/`Float`/`Rational`. Float-only
  `imaginary` programs (`integrate`, `kahan`, `x2n1`) are therefore omitted;
  `bernouilli` uses explicit reduced `(numerator, denominator)` pairs in place of
  `Data.Ratio`.
- **No module system / no type classes.** Every benchmark is a single file that
  inlines the helpers it needs (`map`, `foldr`, `toString`, `++`, …). There is
  no shared prelude and no overloading, so e.g. each file defines its own
  `toString`.
- **Input is one command-line argument, not stdin.** Haskell `nofib` pipes a
  `.stdin` workload; PureLang reads `argv[1]` via `Act (#(cline_arg) " ")`. Each
  benchmark therefore takes a single integer (or, for `gen_regexps`, a string)
  that scales the work. `sizes.cfg` records the value for each run mode.
- **`case` is strict and patterns are shallow** (no nested constructor patterns,
  must be exhaustive). List comprehensions and guards are desugared by hand to
  `concatMap`/`filter`/`if`, and nested patterns are split into separate `case`s.
  A few programs that hinge on Haskell's lazy *irrefutable* patterns are adapted
  (see notes below) or omitted (`wheel-sieve2`).

Output strings are kept close to the originals but are not byte-identical to
GHC's (we print results directly rather than hashing them); what matters is that
the values computed are correct.


## The benchmarks

| Subset    | Name           | Arg | Computes | Notes vs. GHC nofib |
|-----------|----------------|-----|----------|---------------------|
| imaginary | `tak`          | n   | `tak (3n) (2n) n` (Takeuchi) | n=8 ⇒ classic `tak 24 16 8` |
| imaginary | `rfib`         | n   | `nfib n` | original returns `Double`; here integer `nfib` |
| imaginary | `exp3_8`       | n   | `3^n` via Peano numerals | n=8 ⇒ 6561; `slow` needs a large heap |
| imaginary | `primes`       | n   | n-th prime by `iterate`-filter | faithful (the slow sieve) |
| imaginary | `queens`       | n   | # solutions to n-queens | faithful (LML formulation) |
| imaginary | `gen_regexps`  | rx  | size of a generalised-regexp expansion | prints char count (RJE variant); `slow` needs a large heap |
| imaginary | `digits-of-e1` | n   | n digits of e (continued fraction) | prints the digits |
| imaginary | `digits-of-e2` | n   | n digits of e (factorial base) | strict `case` ⇒ explicit base case; 2n-budget keeps it correct |
| imaginary | `wheel-sieve1` | n   | n-th prime (lazy wheel sieve) | first multiple kept lazy to untie the `primes` knot |
| imaginary | `paraffins`    | n   | radical / paraffin counts ≤ n | `Array` memo replaced by a lazy self-referential list |
| imaginary | `bernouilli`   | n   | n-th Bernoulli number | `Data.Ratio` replaced by `(num, den)` pairs |
| spectral  | `primetest`    | p   | is `2^p − 1` (Mersenne) prime? | Miller-Rabin (fixed witnesses) modular exponentiation |
| spectral  | `sorting`      | n   | run 8 sorts on n integers, check agreement | input generated by LCG (no file I/O) |

**Not ported:** `integrate`, `kahan`, `x2n1` (floating point); `wheel-sieve2`
(relies on a lazy infinite `roll`/`dropWhile` spiral that PureLang's strict
`case` cannot express cleanly — `wheel-sieve1` covers the same algorithm family);
the entire `real` subset.


## Running

```bash
./run.py                 # compile, run and check every benchmark at `norm`
./run.py --mode fast     # the quick workloads (good for a smoke test)
./run.py --mode slow     # the heavy workloads
./run.py -k tak -k primes  # only benchmarks whose name matches
./run.py --time --iterations 5   # also report best-of-5 wall-clock time
./run.py --no-compile    # reuse already-built executables
./run.py --list          # list the benchmarks
```

`run.py` compiles each benchmark through the existing `../../Makefile`
(`PUREOPT` is forwarded, so you can pass optimisation flags via `--pureopt`),
runs the resulting executable from `../../out/...`, and diffs its stdout against
the recorded reference for the chosen mode. The CakeML heap size is set with
`--heap` (default 4096 MB; the `slow` modes of `exp3_8` and `gen_regexps` need a
few GB).

### Toolchain note

Compilation needs a working PureCake `pure.S` and a **compatible** CakeML `cake`.
The serialised-AST format shared between them changes over time, so a freshly
`make download`ed `pure.S` and a freshly downloaded `cake` release may not agree.
For reliable results build `pure.S` locally from `../../../compiler/binary` (it
will then match a same-vintage CakeML checkout). This is a pre-existing property
of `examples/`, not specific to this suite.


## Regenerating reference outputs

The `*.stdout` files are produced by running each benchmark at the corresponding
size. If you change a benchmark, rebuild it and overwrite its references, e.g.:

```bash
make -C ../.. benchmark/nofib/imaginary/tak/tak.exe
CML_HEAP_SIZE=4096 ../../out/benchmark/nofib/imaginary/tak/tak.exe 9 \
  > imaginary/tak/tak.stdout
```


## Adding a benchmark

1. `mkdir <subset>/<name>` and write a self-contained `<name>.hs` taking one
   command-line argument (copy the I/O helper block from any existing benchmark).
2. Add `sizes.cfg` with `fast`/`norm`/`slow` arguments (target roughly
   <0.2 s / 1–2 s / 5–10 s).
3. Build it and record the three `*stdout` references (see above).
4. `./run.py -k <name>` to check it passes.


## Relationship to `../benchmark.py`

`../benchmark.py` measures the effect of individual PureCake *optimisation flags*
(timing and heap allocation) on a handful of programs. This suite is
complementary: a broader, output-checked set of programs for tracking
whole-compiler correctness and performance, in the style of GHC `nofib`. Per the
`nofib` reporting rules, raw per-benchmark results should be published before any
derived summary figures.
